DATA FLOW
•	Authentication Flow 
1.	Student or Instructor submits login credentials (SAC or IMC → PSS). 
2.	PSS validates credentials against database. 
3.	If valid, PSS issues JWT token. 
4.	Client stores token and uses it for authenticated requests. 
Data Movement:
Client -> PSS -> Database -> PSS -> Client

•	Room Creation and Assignment Flow
1.	Instructor creates a room (IMC → PSS). 
2.	PSS stores room record in database. 
3.	Instructor assigns students manually OR generates room code. 
4.	PSS stores: 
o	Assigned student records 
o	Room code (if generated) 
Data Movement:
IMC -> PSS -> Database
If room code is used for registering students to the room:
1.)	Student enters room code (SAC → PSS). 
2.)	PSS validates code. 
3.)	If valid, enrollment record is created. 
Data Movement:
SAC -> PSS -> Database







•	Room Discovery & Join Flow 
1.	Student logs into SAC. 
2.	SAC requests list of available/assigned rooms (SAC -> PSS). 
3.	PSS retrieves rooms from database. 
4.	Rooms appear as orb components in SAC dashboard. 
When student joins:
5.	Student clicks Join. 
6.	SAC sends join request to PSS. 
7.	PSS records: 
•	Join timestamp 
•	Participation record 
8.	PSS notifies IMC in real time. 
Data Movement:
SAC → PSS → Database
PSS → IMC (real-time update)

•	Controlled Room Start Flow 
1.	Instructor presses Start (IMC → PSS). 
2.	PSS updates room status to Countdown. 
3.	PSS broadcasts Countdown event to all joined SAC clients. 
4.	Countdown completes. 
5.	PSS updates room status to Active. 
6.	PSS broadcasts SessionStarted event. 
7.	SAC activates monitoring modules. 
Data Movement:
IMC → PSS → Database
PSS → SAC (real-time broadcast)





•	Monitoring & Violation Flow 
1.	SAC detects behavioral or environment event. 
2.	SAC evaluates severity locally. 
3.	SAC sends monitoring event to PSS in real time. 
4.	PSS stores event in database. 
5.	PSS processes event for violation classification. 
6.	If necessary, PSS sends live alert to IMC dashboard. 
Data Movement:
SAC → PSS → Database
PSS → IMC (real-time alert)
•	Connection Tracking Flow 
1.	SAC connects to room hub. 
2.	PSS tracks connection state internally. 
3.	If SAC disconnects: 
o	PSS updates connection status in database. 
o	PSS notifies IMC dashboard. 
Data Movement:
SAC ↔ PSS
PSS → Database
PSS → IMC
•	Room End & Reporting Flow 
1.	Instructor presses End (IMC → PSS). 
2.	PSS updates room status to Ended. 
3.	PSS broadcasts SessionEnded event to SAC clients. 
4.	SAC stops monitoring. 
5.	PSS computes final risk summaries. 
6.	Instructor requests report (IMC → PSS). 
7.	PSS retrieves stored data and sends report. 
Data Movement:
IMC → PSS → Database
PSS → SAC (room end)
PSS → IMC (report data)

•	OVERALL SYSTEM FLOW SUMMARY
Enrollment & Setup:
IMC ↔ PSS ↔ Database
SAC ↔ PSS ↔ Database
Real-Time Monitoring:
SAC ↔ PSS ↔ IMC
Data Storage:
All persistent data flows into Database through PSS only.
SAC and IMC never communicate directly.
All communication passes through PSS.

DATABASE TABLES
Users 
Purpose: Stores both students and instructors.
Fields:
•	Id (PK)
•	Email (Unique)
•	PasswordHash
•	Role (Student / Instructor)
•	CreatedAt
Notes:
•	Single table for both roles. 
•	Role controls system access.


Rooms 
Purpose: Represents one monitoring room.
Fields:
•	Id (PK)
•	SubjectName
•	InstructorId (FK → Users.Id)
•	Status (Pending / Countdown / Active / Ended)
•	EnrollmentCode (nullable, unique)
•	CodeExpiry (nullable)
•	CreatedAt
Notes:
•	EnrollmentCode is optional. 
•	If null → manual-only room. 
•	If generated → students may enroll using code.


SessionAssignments 
Purpose: Tracks manually assigned students.
Fields:
•	Id (PK)
•	SessionId (FK → Rooms.Id)
•	StudentId (FK → Users.Id)
•	AssignedAt
Notes:
•	Only used for manual assignment. 
•	Defines expected students. 
•	One student per room (unique constraint recommended). 



Room Enrollments 
Purpose: Tracks students who enrolled using code.
Fields:
•	Id (PK)
•	SessionId (FK → Rooms.Id)
•	StudentId (FK → Users.Id)
•	EnrollmentSource (Manual / Code)
•	EnrolledAt
Notes:
•	Manual assignment may automatically create an enrollment record. 
•	If student joins via code → EnrollmentSource = Code. 
•	This table defines who is officially enrolled in room. 
Important:
You may merge SessionAssignments into this table by using EnrollmentSource.
If simplified design is preferred, use only this table instead of both.’


SessionParticipants 
Purpose: Tracks actual participation (who joined and monitored).
Fields:
•	Id (PK)
•	SessionId (FK → Rooms.Id)
•	StudentId (FK → Users.Id)
•	JoinedAt
•	DisconnectedAt (nullable)
•	ConnectionStatus (Connected / Disconnected)
•	FinalRiskLevel (Safe / Suspicious / Cheating, nullable)
Notes:
•	Enrollment ≠ Participation. 
•	A student may be enrolled but never join. 
•	This separation is important for reporting. 

MonitoringEvents 
Purpose: Stores all detected monitoring events.
Fields:
•	Id (PK)
•	SessionId (FK → Rooms.Id)
•	StudentId (FK → Users.Id)
•	EventType (ALT_TAB, PROCESS, CLIPBOARD, IDLE, VM, etc.)
•	SeverityScore
•	Timestamp
Notes:
•	Every SendMonitoringEvent creates one row. 
•	Used for timeline and risk computation.





RiskSummaries (Optional but Recommended) 
Purpose: Precomputed room results per student.
Fields:
•	Id (PK)
•	SessionId (FK → Rooms.Id)
•	StudentId (FK → Users.Id)
•	TotalViolations
•	TotalSeverityScore
•	RiskLevel (Safe / Suspicious / Cheating)
•	ComputedAt
Notes:
•	Improves report performance. 
•	Can be generated when room ends.

DETECTION CONFIGURATION FEATURE (ADDED)

Instructor Monitoring Console (IMC) – Detection Settings
• Instructors may configure monitoring behavior per room before the room starts.
• Settings are editable only when room status is Pending.
• Settings become locked once room becomes Active.
• Configurable options include:
    - Enable / Disable Clipboard Monitoring
    - Enable / Disable Process Blacklist Detection
    - Enable / Disable Idle Detection
    - Configure Idle Threshold (seconds)
    - Enable / Disable Focus Detection (Alt+Tab / Window Switch)
    - Enable / Disable Virtualization Detection
    - Select Strict Mode (higher severity weighting)

Secure Assessment Client (SAC) – Behavior Adaptation
• Upon joining a room, SAC requests the room’s detection settings from the server.
• SAC activates only the enabled monitoring modules.
• Idle detection timer follows the configured threshold value.
• Disabled modules remain inactive for the duration of the room.

Proctoring Room Server (PSS) – Detection Control Management
• Stores detection configuration per room.
• Validates that only instructors can modify detection settings.
• Prevents modification once room status is Active.
• Provides detection settings to SAC upon request.
• Applies strict mode weighting during risk computation if enabled.
